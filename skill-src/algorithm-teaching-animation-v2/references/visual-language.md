# Visual Language

本文件定義 `algorithm-teaching-animation` skill 中的視覺語言原則。

此 skill 的視覺設計目標，不是追求花哨或 cinematic 效果，而是讓演算法的狀態變化、資料結構操作、控制流程、不變量與局部到全域的推進過程變得清楚、可追蹤、可教。

本文件的重點是：

- 穩定視覺語義
- 明確畫面焦點
- 可追蹤的結構變化
- 不干擾理解的視覺節奏

## 核心原則

所有視覺設計都應遵守以下原則：

- 教學清晰度優先於裝飾風格  
  若某個視覺效果會讓理解變差，即使看起來更酷，也不應使用。

- 穩定優先於炫技  
  畫面中的資料結構、角色與視覺語義應盡量穩定，避免頻繁重排與無意義動態。

- 焦點必須明確  
  每一個 beat 都應讓觀眾知道現在應該先看哪裡。

- 語義要固定  
  顏色、形狀、標記、指標位置等應有固定意義，不可隨意變更。

- 變化必須可感知  
  重要狀態改變要有足夠清楚的視覺訊號與停留時間。

- 畫面密度必須受控  
  一次不要讓太多元素同時搶注意力。

## 視覺語義的角色分層

視覺語言的核心，是讓觀眾分得出不同角色。

至少應能清楚區分以下角色：

- 預設狀態
- 目前焦點
- 正在比較的元素
- 正在更新或被覆寫的元素
- 已完成或已固定的元素
- 指標、游標或邊界
- 已訪問或已處理的節點 / 狀態
- 當前子問題、活動區間或當前 cell
- 已被排除的區間或元素

如果觀眾無法一眼分辨這些角色，視覺語言就不夠成功。

## 顏色使用原則

### 顏色應服務語義，不應只是裝飾

顏色的用途是讓觀眾快速辨認角色與狀態。

例如：

- 預設狀態：穩定、低干擾的基底色
- 目前焦點：高可見度 highlight 色
- 已完成狀態：穩定完成色
- pointer：與資料元素有區別但不搶主體注意力
- 錯誤或衝突：僅在真的需要時才用強烈對比色

### 同一語義應維持同一顏色

如果 `YELLOW` 在前半段代表「目前比較中」，後半段就不應突然改成代表「已完成」。

### 不要讓太多顏色同時競爭

若同一畫面上同時有太多高彩度重點色，觀眾會失去主要焦點。

原則：

- 主要焦點色數量應有限
- 次要狀態應使用較弱對比
- 預設元素應能退到背景，不與焦點競爭

## Highlight 原則

highlight 的目的，是把觀眾的注意力導向「現在最重要的地方」。

### highlight 適合用來表達：

- 正在比較的元素
- 當前要檢查的位置
- 正在計算的 cell
- 活躍中的區間或節點

### highlight 不應濫用於：

- 純裝飾閃爍
- 所有元素都一起亮
- 沒有教學意義的視覺躁動

### highlight 的使用建議

- 一次最多只突出少數主要焦點
- 焦點轉移時應清楚取消舊焦點
- 不要讓上一拍的 highlight 殘留，造成語義混亂

## 已完成 / 已固定狀態

對演算法教學來說，「哪些部分已經解決」通常非常重要。

因此已完成或已固定狀態應有穩定表示，例如：

- 顏色改為完成色
- 邊框變化
- opacity 或飽和度改變
- 加上完成標記

原則：

- 已完成狀態要能與「目前焦點」區分
- 已完成狀態通常應降低動態，讓觀眾知道它已不再是本拍重點

## 已排除區間 / 已失效範圍

對 binary search、sliding window、partition 類任務來說，「哪些部分已不再屬於有效範圍」同樣重要。

已排除區間應有穩定表示，例如：

- 顏色轉為較淡或較灰的 excluded style
- 降低飽和度或存在感
- 明確區分於「目前焦點」與「已完成」狀態

原則：

- 被排除不等於已完成
- 被排除區間的視覺語義應在整支動畫中保持一致
- 不要用單純取消 highlight 來兼任排除語義

## 指標與邊界

指標與邊界是演算法教學中非常重要的視覺角色。

常見包括：

- `i`、`j`
- `left`、`right`
- `mid`
- 當前 row / col
- partition boundary
- frontier boundary

### 指標設計原則

- label 必須清楚可讀
- 指標位置應穩定且容易追蹤
- 不同指標若同時存在，應能區分角色
- 指標本身不應比主要資料結構更搶眼

### 指標移動原則

- 移動必須對應演算法語義
- 移動距離再大，也應讓觀眾看得出「從哪裡移到哪裡」
- 若一個指標的角色已結束，應適時移除

## 版面穩定性

### 位置應盡量穩定

資料結構的位置穩定，觀眾才容易建立心智模型。

例如：

- 陣列元素的基準位置應固定
- DP 表格不應反覆重排
- 樹與圖的節點布局應盡量穩定

### 不要為了局部更新重排整體結構

除非演算法本身需要，否則不要讓一次局部操作導致整個畫面大幅重新布局。

### 群組與空間關係要可預期

觀眾應能逐步熟悉畫面中的結構位置，而不是每拍重新適應。

## 焦點控制

每個 beat 都應該回答：

- 現在主角是誰？
- 哪裡是第一注意點？
- 哪裡是第二注意點？
- 哪些元素現在只是背景？

### 焦點控制建議

- 主焦點通常只有一個
- 次焦點可以存在，但不應搶過主焦點
- 非主焦點元素可透過顏色、透明度、靜止狀態退到背景

### 常見錯誤

- 同一拍中同時高亮太多元素
- 指標、陣列、文字說明全部都在搶 attention
- 完成狀態和目前焦點顏色太接近

## 畫面密度

### 一次不要呈現太多新的資訊

若同一拍同時出現：

- 新資料結構
- 多個 pointer
- 多組 highlight
- 大量說明文字

觀眾通常很難有效吸收。

### 控制畫面密度的方法

- 一次只引入少數新元素
- 已不重要的元素降低存在感
- 把複雜內容拆成多個 beat
- 只保留教學上真的需要的標記與標籤

## 文字與標籤

### 文字應服務理解，不應佔據主舞台

標籤與說明文字應用來補充理解，而不是搶走主體資料結構的注意力。

### 文字內容建議

- 短
- 直白
- 與當前 beat 緊密相關

### 不建議的文字用法

- 長段敘述直接塞進畫面
- 每一拍都放太多說明文字
- 使用和觀眾程度不匹配的過度抽象文字

## 字幕區

字幕是選配。若使用者明確要求字幕，字幕應被視為穩定的畫面區域，而不是臨時漂浮在主內容上。

原則：

- 預留固定的 subtitle-safe area
- 字幕不應遮住主體資料結構
- 字幕區應與主畫面保持穩定空間關係
- 字幕字數應受控，避免把畫面底部擠滿

## 旁白同步

完整 workflow 預設有旁白音訊。視覺焦點應配合當前 beat 的旁白，而不是讓旁白追趕已經消失的畫面。

原則：

- 每段旁白開始時，畫面焦點應已經可辨識
- 若視覺動作先完成，畫面應維持可理解的狀態直到旁白結束
- 若旁白長到拖慢畫面，優先回修 `voiceover.md` 或拆分 beat
- 不要為了配合旁白，在畫面中加入 trace 沒有支撐的新演算法事件

## 動畫節奏

雖然 trace 不記錄動畫細節，但視覺語言仍必須有可讀節奏。

### 節奏原則

- 重要變化應有足夠停留時間
- 焦點轉移不應太 abrupt
- 交換、覆寫、pointer 移動等關鍵事件應讓觀眾能看清
- 一拍結束時，觀眾應有短暫時間消化結果

### v2 的實務方向

在 v2 中，節奏應優先追求：

- 看得懂
- 跟得上
- 不誤解
- 聽到的旁白與看到的狀態同步

而不是追求電影式速度變化或高度風格化 timing。

## 不變量的視覺化

不變量是這個 skill 的重要教學重點，因此應盡量視覺化。

做法可以包括：

- 用穩定完成色標出已固定區域
- 用區間標示目前仍有效的搜尋範圍
- 用持續存在的 pointer 表示控制流程位置
- 用布局穩定性呈現已建立的結構關係
- 用一致的標記表示已處理 / 未處理狀態

若不變量只能靠旁白理解，通常還可以再加強視覺呈現。

## 各類任務的視覺重點

### Sorting / Array Tasks

應優先讓觀眾看見：

- 比較對象
- 交換或覆寫
- 本輪已固定位置
- pointer 或索引移動

### Binary Search / Pointer-heavy Tasks

應優先讓觀眾看見：

- 當前有效區間
- `left` / `right` / `mid`
- 哪一半被排除

### Graph Traversal

應優先讓觀眾看見：

- 當前節點
- visited 與未 visited 的區別
- frontier 如何擴張或收縮
- frontier 與 current node 的區別

### Dynamic Programming

應優先讓觀眾看見：

- 當前 cell
- 依賴來源
- 已完成區域
- 表格填寫順序

### Trees / Recursive Structures

應優先讓觀眾看見：

- 當前路徑
- 正在比較或插入的位置
- 哪個子樹仍在處理範圍內

## 常見錯誤

### 錯誤 1：顏色只有好看，沒有語義

問題：

- 觀眾記不住每個顏色代表什麼

正確方向：

- 每個顏色都要綁定清楚角色

### 錯誤 2：所有重要元素都一樣亮

問題：

- 沒有主焦點

正確方向：

- 讓主焦點最突出，其餘角色適度退後

### 錯誤 3：畫面因局部操作而整體跳動

問題：

- 觀眾心智模型被破壞

正確方向：

- 保持整體布局穩定，只讓必要局部變化被看見

### 錯誤 4：pointer 太搶眼

問題：

- 指標本來是輔助，卻比主體更吸睛

正確方向：

- 指標應清楚但不應壓過資料結構

### 錯誤 5：每拍塞太多資訊

問題：

- 觀眾無法知道該先看哪裡

正確方向：

- 收斂焦點，拆分 beat

## 最小檢查清單

在設計或 review 畫面時，至少確認：

- 主焦點是否清楚？
- 顏色語義是否一致？
- 已完成與目前焦點是否容易區分？
- pointer 是否易於追蹤？
- 畫面是否穩定？
- 是否有太多同時競爭的高亮元素？
- 不變量是否有被視覺化？
- 文字是否真的幫助理解，而不是干擾？

  決定本次動畫應強調哪些結構與重點

- `teaching-script.md`  
  決定每個 beat 的畫面焦點與教學說明

- `trace-schema.md`  
  決定哪些事件會出現在 trace 中

- `generated_algo_scene.py`  
  依據這些視覺語言原則，把 trace 轉成清楚可教的動畫
